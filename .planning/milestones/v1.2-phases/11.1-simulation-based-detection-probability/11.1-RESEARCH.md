**Researched:** 2026-03-30

## Summary

Phase 11.1 replaces the KDE-based `DetectionProbability` class with a simulation-based detection probability built from a dedicated SNR-only injection campaign. The current `DetectionProbability` (in `detection_probability.py`) builds 3D and 4D KDEs from detected/undetected event DataFrames, then pre-computes `RegularGridInterpolator` look-ups. It is currently disabled via `_DEBUG_DISABLE_DETECTION_PROBABILITY = True` because it was identified as the root cause of H0 posterior bias (debug session confirmed).

The phase has three major work streams: (1) an injection campaign that draws EMRI events per h value, computes SNR only (no Fisher matrix), and stores raw results to CSV; (2) a new `SimulationDetectionProbability` class that loads injection CSVs, applies an SNR threshold at evaluation time, and builds interpolation grids; and (3) integration into Pipeline B by replacing all ~15 call sites in `bayesian_statistics.py`, removing the debug flag, and deleting the old KDE class.

**Primary recommendation:** Build the injection campaign as a new CLI mode (`--injection_campaign`) and a new SLURM sbatch script, reusing the existing `Model1CrossCheck` sampler and `ParameterEstimation.compute_signal_to_noise_ratio()` but skipping Fisher matrix computation entirely. The new `SimulationDetectionProbability` class should provide the same method signatures as the old class (`detection_probability_with_bh_mass_interpolated`, `detection_probability_without_bh_mass_interpolated`) to minimize changes in `bayesian_statistics.py`.

## User Constraints (from CONTEXT.md)

### Locked Decisions
- **D-01:** Build full 5D injection data: store (z, M, phi, theta, SNR, h_inj) per injected event. P_det grid can be constructed at any dimensionality from this raw data.

### Pattern 1: Injection Campaign Event Loop
**What:** SNR-only simulation loop that draws events from the population model for a given h value, computes d_L(z, h), generates waveform, computes SNR, and stores raw results.
**When to use:** When building the injection campaign code.
**Key insight:** The existing `data_simulation()` in `main.py` already does event sampling + SNR computation + CSV storage. The injection campaign is a simplified version that:
1. Takes an explicit `h` value (not using the fiducial h=0.73)
2. Calls `dist(z, h=h_inj)` to compute d_L for the candidate h
3. Skips the Fisher matrix / CRB computation entirely
4. Skips the `save_not_detected()` path (stores ALL events, detected or not)
5. Does NOT need galaxy catalog lookups (events are drawn from the population model directly)

**Critical d_L(z, h) dependency:** In `ParameterSpace.set_host_galaxy_parameters()` (line 148), `dist(host_galaxy.z)` uses the default `h=0.73`. For the injection campaign, the luminosity distance MUST be computed with the candidate h: `dist(z, h=h_inj)`. This means the injection loop cannot use `set_host_galaxy_parameters()` directly -- it must set `luminosity_distance.value = dist(z, h=h_inj)` explicitly.

**Example:**
```python
# Injection campaign loop (simplified)
for sample in cosmological_model.sample_emri_events(batch_size):
    parameter_estimation.parameter_space.randomize_parameters(rng=rng)
    # Set M and sky position from population model
    parameter_estimation.parameter_space.M.value = sample.M
    parameter_estimation.parameter_space.phiS.value = rng.uniform(0, 2 * np.pi)
    parameter_estimation.parameter_space.qS.value = np.arccos(rng.uniform(-1, 1))
    # Compute d_L with the candidate h value
    parameter_estimation.parameter_space.luminosity_distance.value = dist(
        sample.redshift, h=h_inj
    )
    try:
        snr = parameter_estimation.compute_signal_to_noise_ratio()
    except (Warning, ParameterOutOfBoundsError, RuntimeError, ...):
        continue  # skip failed waveforms
    # Store ALL events (detected and undetected)
    store_injection_result(z=sample.redshift, M=sample.M,
                          phi=parameter_estimation.parameter_space.phiS.value,
                          theta=parameter_estimation.parameter_space.qS.value,
                          SNR=snr, h_inj=h_inj)
```

### Pattern 2: SimulationDetectionProbability Class
**What:** Loads injection CSVs, applies SNR threshold at evaluation time, builds interpolation grids.
**When to use:** At evaluation time in Pipeline B.
**Key design:**
- Constructor takes path to injection data directory and SNR threshold
- Loads all injection CSVs (possibly multiple per h value for incremental accumulation)
- For each h grid point: bins events into (z, M, phi, theta) grid, computes fraction with SNR >= threshold
- Builds `RegularGridInterpolator` for each h value
- At evaluation time: interpolates between h grid points
- Must provide same method signatures as old class for drop-in replacement

**Interface (drop-in for old class):**
```python
class SimulationDetectionProbability:
    def __init__(
        self,
        injection_data_dir: str,
        snr_threshold: float,
        h_grid: list[float] | None = None,  # auto-detect from files if None
    ) -> None: ...

    def detection_probability_with_bh_mass_interpolated(
        self,
        d_L: float | npt.NDArray[np.float64],
        M_z: float | npt.NDArray[np.float64],
        phi: float | npt.NDArray[np.float64],
        theta: float | npt.NDArray[np.float64],
    ) -> float | npt.NDArray[np.float64]: ...

    def detection_probability_without_bh_mass_interpolated(
        self,
        d_L: float | npt.NDArray[np.float64],
        phi: float | npt.NDArray[np.float64],
        theta: float | npt.NDArray[np.float64],
    ) -> float | npt.NDArray[np.float64]: ...
```

**Critical change from old class:** The old class does NOT depend on h. The new class MUST be h-aware because P_det depends on h. This means `single_host_likelihood()` must pass the current h value to the P_det lookup. The interpolators must either:
(a) Be rebuilt for each h value (slow), or
(b) Include h as an interpolation dimension (preferred -- 5D or 6D interpolator), or
(c) Pre-build interpolators per h grid point and interpolate between them.

**Recommendation:** Option (c) -- build separate interpolators per h grid point, then linearly interpolate the P_det values between the two bracketing h grid points. This is simpler than a full 5D interpolator and allows incremental addition of h points.

### Pattern 3: d_L to z Coordinate Transformation
**What:** The old DetectionProbability uses d_L as a coordinate axis. But the injection campaign stores events at redshift z. P_det should use z as the native coordinate.
**Why:** The relationship d_L(z, h) depends on h. If we use d_L as a coordinate, we need different d_L grids for different h values. Using z as the coordinate is cleaner because:
1. Events are sampled in z (from population model)
2. The population rate dN/dz is in z
3. In the likelihood integrals, the integration variable is z
4. d_L is computed from z and h at evaluation time

**Implication for method signatures:** The old class accepts d_L. The new class should internally convert d_L to z using `dist_to_redshift(d_L, h=h)`. Or better: modify the call sites to pass z directly (since the likelihood integrals already have z as the integration variable).

### Anti-Patterns to Avoid
- **Building P_det from observed detections:** The old approach uses detected/undetected events from the simulation campaign. This is circular -- P_det should come from a dedicated injection campaign, not from the data being analyzed.
- **Pre-applying SNR threshold in the CSV:** D-03 requires storing raw SNR values. The threshold is applied at evaluation time.
- **Single monolithic CSV:** Use per-h-value files for incremental accumulation and parallel I/O.

## Common Pitfalls

### Pitfall 1: Forgetting h-dependent d_L in injection campaign
**What goes wrong:** Using `dist(z)` (default h=0.73) instead of `dist(z, h=h_inj)` for the injection campaign.
**Why it happens:** `ParameterSpace.set_host_galaxy_parameters()` hardcodes the default h. The injection campaign must bypass this.
**How to avoid:** Explicitly set `luminosity_distance.value = dist(z, h=h_inj)` in the injection loop.
**Warning signs:** All h values produce identical P_det grids.

### Pitfall 2: Empty bins producing NaN or 0 P_det
**What goes wrong:** Bins with zero injected events produce division by zero (0 detected / 0 total).
**Why it happens:** 10,000 events spread over 4 dimensions can leave many bins empty, especially at high z or extreme angles.
**How to avoid:** Use `fill_value=0.0` in RegularGridInterpolator for out-of-bounds queries. For empty bins, assign P_det = 0 (conservative -- if no events land there, detection probability is effectively zero for that parameter combination). Alternatively, use smoothing or reduce grid resolution.
**Warning signs:** NaN values in the interpolator grid.

### Pitfall 3: Coordinate mismatch between injection and evaluation
**What goes wrong:** Injection stores events in (z, M_true) but evaluation needs (d_L, M_z = M_true * (1+z)).
**Why it happens:** The old class uses d_L and M_z as coordinates. The injection campaign samples in (z, M_true).
**How to avoid:** Store both z and d_L in the injection CSV, and both M_true and M_z. Or convert at load time.
**Warning signs:** P_det values are wrong at boundaries.

### Pitfall 4: Multiprocessing pickling of the new class
**What goes wrong:** The new `SimulationDetectionProbability` fails to be pickled for `child_process_init`.
**Why it happens:** `RegularGridInterpolator` objects must be picklable. Large NumPy arrays are fine, but lambda functions or complex objects may fail.
**How to avoid:** Ensure the class is pickle-safe. Test with `pickle.dumps(instance)`.
**Warning signs:** `PicklingError` when spawning worker processes.

### Pitfall 5: Waveform failures reducing effective sample size
**What goes wrong:** Many events fail waveform generation (ParameterOutOfBoundsError, EllipticK errors, timeouts), reducing the effective number of injections below 10,000.
**Why it happens:** The current simulation loop has ~30-50% failure rate for waveform generation.
**How to avoid:** Target 10,000 successful SNR computations, not 10,000 attempts. The loop should count successful events.
**Warning signs:** CSV has significantly fewer rows than expected.

### Pitfall 6: Sky angle sampling in injection vs population model
**What goes wrong:** The population model (`Model1CrossCheck.sample_emri_events()`) returns (M, z, a) but NOT sky angles. Sky angles are randomized independently by `ParameterSpace.randomize_parameters()`.
**Why it happens:** The EMRI rate model dN/dz depends on M and z, not on sky location (which is isotropic).
**How to avoid:** In the injection campaign, sample sky angles uniformly (phi uniform in [0, 2pi], theta from arccos(uniform(-1,1))) and store them. This is what `randomize_parameters()` already does.
**Warning signs:** Anisotropic P_det when it should be approximately isotropic (modulo LISA antenna pattern).

## Grid Construction Details

### Binning Strategy
For P_det(z, M, phi, theta | h):
- **z:** 20-30 bins, linearly spaced from 0 to max_redshift (1.5)
- **M:** 15-20 bins, log-spaced from 10^4.5 to 10^6.0 (matching Model1CrossCheck limits)
- **phi (phiS):** 10-15 bins, linearly spaced from 0 to 2*pi
- **theta (qS):** 10-15 bins, linearly spaced from 0 to pi

Total bins per h value: ~20 * 15 * 10 * 10 = 30,000 bins for 10,000 events = ~0.33 events/bin average. This is sparse.

**Sparse bin mitigation strategies (D-02):**
1. Start with full 4D grid. If too noisy, marginalize over phi and theta (uniform distribution) to get 2D P_det(z, M | h).
2. Use larger bins (fewer bins per dimension).
3. Use kernel smoothing on the binned counts before computing the fraction.

**Recommended starting point:** P_det(z, M | h) with 30 z-bins and 20 M-bins = 600 bins. With 10,000 events, ~17 events/bin average. This should be sufficient for reasonable estimates.

### Interpolation Between h Values (D-06)
For a query at arbitrary h:
1. Find the two bracketing h grid points: h_low <= h <= h_high
2. Evaluate P_det(z, M, ... | h_low) and P_det(z, M, ... | h_high)
3. Linearly interpolate: P_det = (1-t) * P_det_low + t * P_det_high, where t = (h - h_low) / (h_high - h_low)

For h outside the grid range, clamp to the nearest grid point.
