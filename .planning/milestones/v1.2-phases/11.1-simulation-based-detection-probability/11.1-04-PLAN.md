```bash
#!/usr/bin/env bash
# cluster/inject.sbatch -- GPU array job for injection campaign (P_det estimation).
#
# Each array task runs N injection events for a single h value.
# The task-to-h mapping is handled by submit_injection.sh which sets H_VALUE.
#
# Required environment variables (set by submit_injection.sh):
#   RUN_DIR        -- workspace directory for injection output
#   BASE_SEED      -- base random seed (per-task seed = BASE_SEED + SLURM_ARRAY_TASK_ID)
#   INJ_STEPS      -- number of successful injection events per task
#   H_VALUE        -- Hubble constant value for this batch of tasks

#SBATCH --job-name=emri-inject
#SBATCH --partition=dev_gpu_h100
#SBATCH --gres=gpu:1
#SBATCH --time=00:30:00
#SBATCH --ntasks=1
```

**Environment setup**: identical to simulate.sbatch (source modules.sh + venv activate, validate RUN_DIR/BASE_SEED).

**Additional validation**: check H_VALUE and INJ_STEPS are set.

**Seed derivation**: `TASK_SEED=$((BASE_SEED + SLURM_ARRAY_TASK_ID))` (same pattern).

**Working directory setup**: same pattern as simulate.sbatch:
```bash
cd "$PROJECT_ROOT"
mkdir -p "$RUN_DIR/simulations/injections"
ln -sfn "$RUN_DIR/simulations" "$PROJECT_ROOT/simulations"
```

**Run command**:
```bash
python -m master_thesis_code "$RUN_DIR" \
    --injection_campaign \
    --h_value "$H_VALUE" \
    --simulation_steps "$INJ_STEPS" \
    --simulation_index "$SLURM_ARRAY_TASK_ID" \
    --seed "$TASK_SEED" \
    --use_gpu \
    --log_level INFO
```

**Summary output**: echo task ID, h value, seed, steps at start and completion.

Create `cluster/submit_injection.sh` that submits injection campaign jobs for multiple h values.

**Usage:**
```bash
submit_injection.sh --tasks_per_h 20 --steps 500 --seed 12345
# Runs 20 tasks * 500 events = 10,000 events per h value
# 7 h values * 20 tasks = 140 total SLURM tasks
```

**Arguments:**
- `--tasks_per_h N` (required): Number of array tasks per h value
- `--steps N` (required): Successful injection events per task
- `--seed N` (required): Base random seed
- `--h_values "0.60,0.65,..."` (optional, default: `"0.60,0.65,0.70,0.73,0.80,0.85,0.90"` per D-05)

**Implementation:**
1. Parse arguments (same pattern as submit_pipeline.sh)
2. Set up RUN_DIR using bwHPC workspace or fallback to `$HOME/emri-runs/injection-$(date +%Y%m%d-%H%M%S)`
3. `mkdir -p "$RUN_DIR/simulations/injections"`
4. Split h_values string by comma into array
5. For each h value in the array:
   - Submit `cluster/inject.sbatch` with `--array=0-$((TASKS_PER_H - 1))`
   - Export: `RUN_DIR`, `BASE_SEED`, `INJ_STEPS=$STEPS`, `H_VALUE=$h`
   - Set `--output` and `--error` to `$RUN_DIR/logs/inject_h_${h_label}_%A_%a.out` / `.err`
   - Capture JOB_ID for summary
6. Print summary: number of h values, tasks per h, total tasks, expected events, RUN_DIR

**Seed isolation between h values**: Use `BASE_SEED + h_index * 10000` as the per-h base seed so different h values get non-overlapping seed ranges. E.g., for 20 tasks_per_h:
- h=0.60: seeds 12345+0 to 12345+19
- h=0.65: seeds 22345+0 to 22345+19 (12345 + 1*10000)
- etc.

**Key difference from submit_pipeline.sh**: No dependency chaining (inject jobs are independent). No merge or evaluate steps -- those happen locally after rsync.
