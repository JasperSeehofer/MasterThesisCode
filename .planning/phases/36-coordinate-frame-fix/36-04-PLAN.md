    def setup_4d_galaxy_catalog_balltree(self) -> None:
        """Build the 5-D host-assignment BallTree (sky chord + z + log M).

        The sky sub-space uses spherical Cartesian embedding via
        `_polar_to_cartesian(θ, φ)` so that chord-length on the unit sphere
        is the sky metric — avoiding the COORD-02b flat-metric bug that
        collapsed equatorial points to a corner of the flat (φ/2π, θ/π)
        square. Redshift and log-mass axes are linearly normalized to [0, 1].

        Metric weights (planner's choice per Claude's Discretion in
        .planning/phases/36-coordinate-frame-fix/36-CONTEXT.md D-18):
        sky chord length ∈ [0, 2] + z_norm ∈ [0, 1] + log_M_norm ∈ [0, 1],
        euclidean on ℝ⁵. This gives the sky axes slightly more weight
        than z or M, which matches the physical intuition: two galaxies
        at the same sky position but different z are candidates for the
        same EMRI sky localization; two galaxies at the same z but
        different sky positions are not.

        References:
            COORD-02b fix; .planning/phases/36-coordinate-frame-fix/36-CONTEXT.md D-17, D-18.
            .planning/REQUIREMENTS.md §Coordinate Frame Correctness COORD-02b.
        """
        # Sky sub-space: spherical Cartesian unit vectors (COORD-02b)
        phi = self.reduced_galaxy_catalog[InternalCatalogColumns.PHI_S].values
        theta = self.reduced_galaxy_catalog[InternalCatalogColumns.THETA_S].values
        # Eq. (standard spherical polar); CONTEXT.md D-17.
        sky_xyz = _polar_to_cartesian(theta, phi)  # shape (N, 3), unit vectors

        # Redshift axis: linear normalization to [0, 1]
        redshift_norm = (
            self.reduced_galaxy_catalog[InternalCatalogColumns.REDSHIFT] / self.z_max
        ).values

        # Mass axis: log normalization to [0, 1]
        log_mass = np.log10(self.reduced_galaxy_catalog[InternalCatalogColumns.BH_MASS])
        log_mass_min = np.log10(self.M_min)
        log_mass_max = np.log10(self.M_max)
        mass_norm = ((log_mass - log_mass_min) / (log_mass_max - log_mass_min)).values

        # Combine into a 5-D array: [sky_x, sky_y, sky_z, z_norm, log_M_norm]
        data = np.hstack((sky_xyz, redshift_norm.reshape(-1, 1), mass_norm.reshape(-1, 1)))

        self.catalog_4d_ball_tree = BallTree(data, metric="euclidean")
        _LOGGER.info(
            "5-D BallTree (3 sky Cartesian + z_norm + log_M_norm) built successfully."
        )

  <context>
    Per CONTEXT.md §Claude's Discretion and D-18, the 5-D metric weight is the planner's call. This checkpoint surfaces the choice to the user BEFORE the [PHYSICS] COORD-02b commit lands, since the weight affects which host galaxy gets assigned to a simulated EMRI event.

    Chosen weights:
    - Sky axes: native unit-sphere chord length, ∈ [0, 2]. No rescaling.
    - z_norm: (z / z_max), ∈ [0, 1]. Unchanged from legacy.
    - log_M_norm: (log M − log M_min) / (log M_max − log M_min), ∈ [0, 1]. Unchanged from legacy.

    Implicit relative weights: sky : z : M ≈ 2 : 1 : 1 by maximum range. A 90° separation on the sky contributes chord ≈ √2 ≈ 1.41 to the 5-D distance, which is comparable to a full-range mismatch in z or M.

    Alternatives (NOT chosen):
    - **Equal-weight (explicit rescale)**: divide sky chord by 2 → all three axes in [0, 1]. Simpler, but slightly de-emphasizes sky. Planner REJECTED because sky localization is the primary physical discriminator per the EMRI sky-error-ellipse argument (Fisher CRB).
    - **Sky-dominant (weight 5×)**: multiply sky_xyz by 5 → sky dominates the 5-D distance. REJECTED: makes z and M nearly irrelevant; defeats the point of a 4D-tree (which was introduced to discriminate by (z, M) when the sky CRB is wide).

    Legacy note: The pre-fix code used `phi_norm = φ / 2π` (∈ [0, 1]) and `theta_norm = θ / π` (∈ [0, 1]), so all four axes had equal range [0, 1]. The new implementation gives sky axes range [0, 2], so it is slightly MORE sky-dominant than the legacy code. This is intentional AND is consistent with the physical role of sky coordinates.

    If user wants equal-weighting (recover legacy balance), planner would rescale sky_xyz by 0.5 before hstack. Raise this here rather than silently picking.

    **Execution ordering (Info 6 resolution):** Tasks 1 and 2 implement the planner's default (natural chord-length, sky_xyz used unrescaled). If this checkpoint decides for `equal-weight` (scale sky_xyz by 0.5) or `user-specified` (arbitrary weights), Tasks 1–2 MUST be re-edited BEFORE Task 4 commits (e.g., `sky_xyz = _polar_to_cartesian(...) * weight_factor` in both `setup_4d_galaxy_catalog_balltree` and `find_closest_galaxy_to_coordinates`, plus a corresponding docstring update). The default (`planner-default`) requires no additional edits — Tasks 1–2 as written match it.

    Do NOT commit Task 4 until the checkpoint has resolved. If the checkpoint occurs after a partial stage of Tasks 1–2, unstage and re-edit before proceeding.
