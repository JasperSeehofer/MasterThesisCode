<objective>
Run the production --evaluate pipeline (BayesianStatistics.evaluate) with v2.2 code on current equatorial CRBs to measure the true MAP before any code changes are made.

Purpose: The Phase 40 SC-3 failure (MAP=0.860) was diagnosed using extract_baseline, which lacks the -N log D(h) normalization from Gray et al. (2020) Eq. A.19. BayesianStatistics.evaluate() DOES include D(h) via precompute_completion_denominator. The VERIFY-02 "passing" result was spurious (compared v2.1 archives against themselves). We must measure the true v2.2 MAP before deciding which fix(es) are needed.

Two hypotheses are confirmed by code trace:
- H1: extract_baseline/load_posteriors lacks -N log D(h) → biases MAP to h_max=0.860
- H2: Detection.__init__ reads equatorial qS/phiS from CRBs; v2.2 BallTree is ecliptic → mismatch ~23.4°

The --evaluate path includes D(h), so it tests both H1 and H2 together. The outcome determines the fix path:
- If MAP_evaluate ≈ 0.73 ± 0.01: H2 does not affect production path (BallTree still finds hosts despite mismatch); fix = H1 only (deprecate extract_baseline as MAP reporter)
- If MAP_evaluate ≠ 0.73: H2 is active; fix = CRB frame migration (equatorial → ecliptic) + re-evaluate

Output: Measured --evaluate MAP value; definitive branch decision recorded for Plan 43-02.
</objective>

       ```python
       import json, glob, numpy as np
       files = sorted(glob.glob('simulations/posteriors/h_*.json'))
       results = []
       for f in files:
           data = json.loads(open(f).read())
           h = data['h']
           log_sum = 0.0
           for k, v in data.items():
               if k != 'h' and isinstance(v, list) and v and v[0] > 0:
                   log_sum += np.log(v[0])
           results.append((h, log_sum))
       results.sort()
       map_h = max(results, key=lambda x: x[1])[0]
       print(f"MAP h (from posteriors/) = {map_h}")
       for h, lp in results:
           print(f"  h={h:.3f}  log_L={lp:.2f}")
       ```
