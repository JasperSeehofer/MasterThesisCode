       - The except clause line number in main.py
       - Whether the failure is logged (warning/error message)
       - Whether the failure increments any counter or is silently dropped
       - The `continue` statement that skips recording the failed event

    2. **Source tracing:** For each exception type, trace backward to where it is raised:
       - TimeoutError: likely from the waveform generation timeout decorator
       - ParameterOutOfBoundsError: from parameter validation in ParameterSpace or few library
       - RuntimeError: from few/fastlisaresponse waveform generation (solver divergence)
       - ValueError: from numerical issues (EllipticK, Brent method failures in few)
       - ZeroDivisionError: from edge cases in PSD or inner product computation
       Document the file:line where each exception originates and the physical condition that triggers it.

    3. **Parameter region analysis (code-level):** Based on the code and known waveform generator limitations:
       - High eccentricity (e0 near 0.2): known to cause solver divergence in few
       - Extreme mass ratios (M near boundaries 10^4.5 or 10^6): known edge cases
       - High z (large d_L -> low SNR, but also longer waveforms -> timeout risk)
       - Specific parameter combinations that violate few internal checks
       Document which (z, M, e0) regions are expected to produce which failure types.

    4. **CSV design limitation:** Cite the specific lines where:
       - The results list is initialized (empty)
       - Successful events are appended to results
       - Failed events hit `continue` without being appended
       - The CSV is written from results only
       Document the CSV column list and confirm no failure status field exists.

    5. **Future tracking proposal:** Recommend additions for the next injection campaign:
       - Add a failure CSV or status column recording (z, M, e0, h, exception_type, exception_message)
